# CLAUDE.md 配置规范

## 文件结构

```
~/.claude/
├── CLAUDE.md              # 全局入口：偏好 + @引用列表
└── rules/
    ├── engineering.md     # 工程规范
    ├── git-workflow.md    # Git 工作流
    ├── testing.md         # 测试规范
    └── ...                # 每个主题一个文件
```

项目级规则放项目根目录：
```
项目根/
└── .claude/
    └── rules/
        ├── backend.md
        └── frontend.md
```

## 写法规范

### 具体可验证（最重要）

不写：
- 代码要简洁
- 注意安全

要写：
- 函数不超过 30 行
- 不添加注释到你没有修改的代码行
- commit message 用中文，不超过 50 字
- SQL 查询必须用参数化查询，不拼接字符串

### 每文件一个主题

一个文件只覆盖一个关注点。混合主题时拆开。

### 文件大小

每个规则文件不超过 50 行。超出时拆成子文件或精简重复内容。

### CLAUDE.md 结构

```markdown
# [项目/全局] 指导原则

## [核心偏好1]
- 具体规则

## [核心偏好2]
- 具体规则

@.claude/rules/rule1.md
@.claude/rules/rule2.md
```

CLAUDE.md 本身保持简短（< 30 行），细节放规则文件。

## 层级优先级

1. 用户明确指令（最高）
2. 项目根目录 CLAUDE.md
3. 子目录 CLAUDE.md
4. 全局 ~/.claude/CLAUDE.md（最低）

## 常见错误

| 错误写法 | 正确写法 |
|---------|---------|
| 代码要整洁 | 函数不超过 30 行；变量名用 camelCase |
| 认真测试 | 修复 Bug 前先写能复现的失败测试 |
| 注意性能 | 数据库查询必须有索引；N+1 查询禁止上线 |
| 好好写注释 | 只在非显而易见的逻辑处写注释 |
